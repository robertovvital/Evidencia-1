Analyzing the Halcon case helped me understand that the most important part of a
software project is not writing code, but understanding the business process
well before designing the solution. Translating a narrative description —given in
an interview with the client— into use case, class and activity diagrams forced
me to ask questions that were not initially obvious, such as what happens when
there is no material in stock or who should be able to see the photo-upload
option.

Choosing an agile methodology like Scrum made me reflect on how to break down a
large problem into deliverable, verifiable increments, instead of trying to solve
the entire system at once. Likewise, designing the database model let me see how
the client's requirements (such as not physically deleting an order, but only
"hiding" it) translate into concrete technical decisions, such as using a
logical-deletion flag instead of removing the record.

Overall, this deliverable leaves me with the lesson that analysis and design are
iterative: while drawing each diagram I found things I had to correct in the
others, which reinforced the importance of keeping the data model, the system's
classes and the use cases that each company role will ultimately use consistent
with one another.
